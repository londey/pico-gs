---
description: Spawn an agent team to create and verify digital twin test benches for a component
argument-hint: <component-path> (e.g. components/texture/detail/l1-cache)
---

Create an agent team to plan and implement DT-verified test benches for the
component at $ARGUMENTS.

The component must have both `$ARGUMENTS/rtl/` and `$ARGUMENTS/twin/` directories.
Test benches verify that RTL output binary-matches the digital twin's expected
output for identical stimulus inputs.

## Pre-flight

Before spawning teammates, verify:

1. `$ARGUMENTS/twin/` and `$ARGUMENTS/rtl/` both exist — abort if either is missing
2. Read `$ARGUMENTS/twin/Cargo.toml` to determine the crate name (e.g. `gs-tex-l1-cache`)
3. Read the twin crate's `src/lib.rs` to understand the component's I/O interface,
   Q-format conventions, and data types
4. Check `doc/design/` for the relevant UNIT spec describing this component
5. Check for existing `*_dt.sv` testbenches in `$ARGUMENTS/rtl/tests/` and
   existing `gen_*_vectors.rs` binaries in `$ARGUMENTS/twin/src/bin/` to understand
   established patterns

## Naming conventions

All generated files must follow these patterns:

- Rust vector generators: `gen_<module>_<scenario>_vectors.rs` in `$ARGUMENTS/twin/src/bin/`
- SV testbenches: `tb_<module>_<scenario>_dt.sv` in `$ARGUMENTS/rtl/tests/`

Examples: `gen_l1_cache_wrap_vectors.rs`, `tb_l1_cache_wrap_dt.sv`.

## Teammates

Spawn 2 teammates **sequentially** (rtl-testbench waits for dt-vectors to finish).

### 1. "dt-vectors" — Digital twin test vector generator

Load skill: `claude-skill-rust`

Before writing any code, propose a concrete test plan listing:
- The specific test scenarios to cover (with rationale)
- Edge cases and boundary conditions relevant to this component

**Wait for user approval of the test plan before implementing.**

After approval, implement Rust binaries in `$ARGUMENTS/twin/src/bin/` that generate
hex stimulus and expected-output files for the RTL testbench.

Responsibilities:
- Read the twin crate to understand the component's public API and data formats
- Read the relevant UNIT spec in `doc/design/` for interface details
- Follow existing `gen_*_vectors.rs` patterns in the repo if any exist
- Generate `$readmemh`-compatible hex files: stimulus (inputs) and expected (outputs)
- Ensure `cargo build` and `cargo clippy` pass for the twin crate after changes

### 2. "rtl-testbench" — RTL Verilator testbench

Load skill: `claude-skill-verilog`

Implement SystemVerilog testbenches in `$ARGUMENTS/rtl/tests/` that load the
hex files generated by dt-vectors and verify RTL output matches expected values.

Responsibilities:
- Read the RTL module(s) in `$ARGUMENTS/rtl/src/` to understand ports and timing
- Follow existing `tb_*_dt.sv` patterns in the repo
- Load stimulus and expected hex files via `$readmemh`
- Drive DUT inputs on `@(negedge clk)` (standard convention)
- Compare RTL outputs against expected values with pass/fail counters
- Print clear PASS/FAIL summary with mismatch details
- Ensure `verilator --lint-only -Wall` passes

## Completion

After both teammates finish:
- Wire test benches into `./build.sh --dt-verify` via the integration Makefile
- Run `./build.sh --dt-verify` and confirm all pass
- Fix any integration issues
