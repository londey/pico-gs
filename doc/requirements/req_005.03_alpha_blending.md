# REQ-005.03: Alpha Blending

## Requirement

When the host sets the ALPHA_BLEND field in the RENDER_MODE register to a non-DISABLED mode and submits a triangle for rendering, the system SHALL blend each output fragment color with the existing framebuffer pixel using the selected blend equation before writing the result to the framebuffer.

When the host sets ALPHA_BLEND=DISABLED in RENDER_MODE, the system SHALL overwrite the destination framebuffer pixel with the source fragment color without reading the existing framebuffer contents.

When the host sets ALPHA_BLEND=ADD in RENDER_MODE, the system SHALL add the source fragment color (pre-multiplied by fragment alpha) to the destination framebuffer color, saturating each component at the maximum representable value.

When the host sets ALPHA_BLEND=SUBTRACT in RENDER_MODE, the system SHALL subtract the source fragment color (pre-multiplied by fragment alpha) from the destination framebuffer color, clamping each component to zero.

When the host sets ALPHA_BLEND=BLEND in RENDER_MODE, the system SHALL apply standard Porter-Duff source-over blending per component (R, G, B, A).

## Rationale

Alpha blending enables rendering of transparent and semi-transparent objects by combining source fragment color with existing framebuffer content.
Supporting multiple blend modes (disabled, additive, subtractive, Porter-Duff) enables a range of visual effects without additional draw passes.

Alpha blending operations are performed in Q4.12 fixed-point format.
Destination pixels are read from the RGB565 color tile cache (UNIT-013) and promoted to Q4.12 via MSB-replication before blending.
After blending, the result passes through ordered dithering (REQ-005.10) before RGB565 conversion and write-back to the cache.

**Implementation note:** Blending is implemented via UNIT-010 (Color Combiner), using the `DST_COLOR` source (destination pixel from the color tile cache) in the `(A-B)*C+D` equation.
DST_COLOR is available to all three CC passes; conventionally pass 2 (CC2) is used for alpha blending, but this is not enforced by hardware.
RENDER_MODE.ALPHA_BLEND selects the CC2 equation template; no dedicated alpha blend hardware unit is used.

## Verification Method

**Test:** Execute relevant test suite for alpha blending (VER-024):

- [ ] Set ALPHA_BLEND field in RENDER_MODE register to select blend mode
- [ ] Support DISABLED mode (overwrite destination, no tile buffer read)
- [ ] Support ADD mode (`saturate(dst + src × alpha)`, per component)
- [ ] Support SUBTRACT mode (`saturate(dst − src × alpha)`, clamp to zero per component)
- [ ] Support BLEND mode (standard Porter-Duff source-over)
- [ ] Alpha blend applies per-component (R, G, B, A)
- [ ] Destination pixel is read from the color tile cache (RGB565 promoted to Q4.12) before blending
- [ ] Disable Z_WRITE when rendering transparent objects (Z_TEST still enabled)
- [ ] Verify correct transparency with alpha=0, 0.5, and 1.0

## Notes

User Story: As a firmware developer, I want to blend rendered pixels with framebuffer content based on alpha, so that I can render transparent and semi-transparent objects.

**Early Z compatibility:** Transparent objects are rendered with Z_WRITE=0 (as noted above) but Z_TEST=1.
Early Z-test (REQ-005.04) is compatible with this usage: occluded transparent fragments are correctly rejected by early Z (they would have failed the depth test at any pipeline stage), saving texture and blending work.
Since there is no alpha test (alpha kill) in this pipeline, early Z never incorrectly discards a fragment that would have modified the framebuffer.

**Cache coherency for blended primitives:** When ALPHA_BLEND is non-DISABLED, UNIT-013 supplies the destination pixel from the color tile cache before CC pass 2 executes.
Because both DST_COLOR reads and fragment writes target the same cache, self-overlapping blended primitives within the same draw call are coherent automatically — later fragments read the already-blended value of earlier fragments occupying the same tile.
A cache miss triggers a 16-word burst read from SDRAM to fill the required tile; subsequent accesses to the same tile hit the cache.
The per-pixel SDRAM read-back that existed prior to UNIT-013 is eliminated; all destination access is through the cache.
A flush (FB_CACHE_CTRL.FLUSH_TRIGGER per INT-010) is not required between blended primitives in the same draw call — only at framebuffer swap (see REQ-005.09).

**REQ-028 retired:** REQ-028 was a duplicate functional-format counterpart of this requirement.
It has been retired; REQ-005.03 is now the single canonical alpha blending requirement.
