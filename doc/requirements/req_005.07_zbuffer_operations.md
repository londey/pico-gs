# REQ-005.07: Z-Buffer Operations

## Classification

- **Priority:** Essential
- **Stability:** Stable
- **Verification:** Test

## Requirement

When Z_TEST_EN=1 and a fragment's interpolated depth value is compared against the Z-buffer using the configured Z_COMPARE function, the system SHALL pass the fragment and, when Z_WRITE_EN=1, update the Z-buffer with the new depth value; otherwise the system SHALL discard the fragment without modifying the Z-buffer or color buffer.

When Z_RANGE is configured with Z_RANGE_MIN and Z_RANGE_MAX values, the system SHALL discard any fragment whose interpolated Z value falls outside [Z_RANGE_MIN, Z_RANGE_MAX] before performing the depth comparison.

When Z_WRITE_EN=1 and a fragment passes the depth test, the system SHALL update the Hi-Z metadata for the containing 4×4 tile: if the written depth value is less than the tile's stored minimum depth (min_z), the system SHALL replace min_z with the written depth value.

When Hi-Z metadata for a tile is marked invalid (cleared state), the system SHALL treat any Hi-Z comparison for that tile as passing (i.e., not reject the tile), and the first Z-write to that tile SHALL initialize the metadata with that depth value.

## Rationale

This requirement defines the functional behavior of the Z-buffer operations subsystem, including depth range clipping, Z-buffer write control, and Hi-Z metadata maintenance.
The Z-buffer tile cache (UNIT-006) absorbs most Z read/write traffic; only cache misses and evictions generate SDRAM accesses via the arbiter (UNIT-007).
The Hi-Z metadata (8 DP16KD blocks, UNIT-005) stores a per-tile valid bit and truncated min_z, enabling tile-level rejection in the rasterizer (UNIT-005.05) before any fragments are emitted.
Maintaining min_z on every Z-write is a side-effect of the write path that must not change functional pixel output.

## Verification Method

**Test:** Execute relevant test suite for Z-buffer operations.

- [ ] Depth comparison produces correct pass/fail results for all 8 compare functions (LESS, LEQUAL, EQUAL, GEQUAL, GREATER, NOTEQUAL, ALWAYS, NEVER)
- [ ] When Z_WRITE_EN=1, passing fragments update the Z-buffer with the new depth value
- [ ] When Z_WRITE_EN=0, the Z-buffer is not modified regardless of depth test result
- [ ] When Z_RANGE is configured to a sub-range, fragments outside [Z_RANGE_MIN, Z_RANGE_MAX] are discarded before depth test
- [ ] When Z_RANGE_MIN=0x0000 and Z_RANGE_MAX=0xFFFF, all fragments proceed to depth test (full range passthrough)
- [ ] When Z_WRITE_EN=1 and a fragment passes the depth test, the Hi-Z metadata min_z for the tile is updated if the written depth is less than the stored min_z
- [ ] When Z_WRITE_EN=0, the Hi-Z metadata is not modified regardless of depth test result
- [ ] When a tile's Hi-Z metadata is invalid, the Hi-Z test passes (tile is not rejected) and the first Z-write to that tile initializes min_z

## Notes

The Z-buffer stores 16-bit unsigned depth values (0 = near, 0xFFFF = far) in 4×4 block-tiled layout identical to the framebuffer format (INT-011).
Z-buffer addresses use the same block-tiled address calculation as framebuffer pixel addresses.

The Hi-Z metadata array (UNIT-005) holds one entry per 4×4 tile: a 1-bit valid flag and a truncated 8-bit min_z representing the minimum depth value written to that tile.
The rasterizer (UNIT-005.05) reads this metadata at tile granularity before emitting any fragments; tiles whose stored min_z exceeds the incoming Z threshold are rejected entirely (Hi-Z test, UNIT-005.05).
Hi-Z metadata maintenance on Z-write is a side-effect of the existing Z-write path and must not alter the final framebuffer contents.
