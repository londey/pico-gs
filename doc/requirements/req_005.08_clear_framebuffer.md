# REQ-005.08: Clear Framebuffer

## Requirement

When the host writes the MEM_FILL register (INT-010, index 0x44) with a base address, a 16-bit fill value, and a word count, the system SHALL write that fill value to every specified 16-bit SDRAM word using sequential burst writes, blocking the GPU command pipeline until the fill completes, and SHALL leave framebuffer and Z-buffer words outside the specified range unmodified.

When a Z-buffer clear is issued (MEM_FILL with fill value 0xFFFF targeting the Z-buffer region), the system SHALL reset all Hi-Z metadata entries to the sentinel value (UNIT-005.06) and set all Z-cache uninitialized flags to 1 (UNIT-012), marking every tile as unwritten without requiring SDRAM writes to each individual Z-buffer word at clear time.

When a Z-buffer tile whose Z-cache uninitialized flag is set is first accessed for a Z-read or Z-write, the system SHALL supply the far-plane depth value (0xFFFF) as the effective stored Z, initializing the tile lazily on first use (lazy-fill).

When the host writes FB_CACHE_CTRL.INVALIDATE_TRIGGER (INT-010, index 0x45), the system SHALL drop all valid and dirty bits in the color tile cache (UNIT-013) and reset the color-cache per-tile uninitialized-flag array, marking every color tile as unwritten without requiring SDRAM writes, and SHALL block the GPU command pipeline until the invalidation completes.

When a color tile whose uninitialized flag is set is first written, the system SHALL treat the tile as containing zero-valued pixels (black, RGB565 = 0x0000), initializing it lazily on first write without a cache-fill burst from SDRAM.

## Rationale

A framebuffer clear establishes a known starting state before geometry is rendered each frame.
Without clearing, residual pixel data and depth values from the previous frame produce visual corruption.
The MEM_FILL hardware fill issues direct sequential SDRAM burst writes, achieving maximum fill throughput without rasterizer setup overhead.
Clearing the back-buffer (not the displayed front-buffer) prevents visible tearing during the clear operation.
Because both the color buffer and Z-buffer use 4×4 block-tiled layout with contiguous SDRAM addressing (INT-011), a single MEM_FILL covers any power-of-two surface with no block-aware addressing required.

For Z-buffer clears, a two-phase fast-clear eliminates the need to write 0xFFFF to every Z-buffer word in SDRAM at clear time, reducing clear time from ~2.66 ms (full SDRAM fill) to ~5.12 µs — a ~520× improvement.
The two phases run concurrently: resetting the 8 Hi-Z metadata DP16KD blocks to sentinel values (UNIT-005.06, 512 cycles) and setting all Z-cache uninitialized flags to 1 (UNIT-012, 512 cycles via a separate DP16KD).
Co-locating the uninitialized flag with UNIT-012 (the Z-cache that consumes it for lazy-fill) ensures these two pieces of state are always reset together, eliminating any risk of the flag and cache state going out of sync.
The lazy-fill contract ensures that any tile not yet written by the current frame behaves as if its Z-buffer contents are 0xFFFF (far plane), preserving functional correctness.

For color-buffer clears, the INVALIDATE_TRIGGER in FB_CACHE_CTRL provides a color-side analogue of the Z-cache fast-clear: the cache's per-tile uninitialized-flag array is reset in ~512 cycles, and all cached dirty lines are discarded without a SDRAM writeback.
When the intended clear color is black (0x0000), INVALIDATE_TRIGGER alone suffices — the color cache's lazy-fill with zero means first-touch writes proceed immediately without reading SDRAM.
When a non-zero clear color is required, the driver issues a MEM_FILL to write the fill value to SDRAM, then follows with INVALIDATE_TRIGGER so the cache starts the new frame with a fresh uninitialized-flag state aligned to the newly-filled SDRAM contents.

## Verification Method

**Test:** Issue a MEM_FILL command with a known fill color for the color buffer, then a second MEM_FILL with 0xFFFF for the Z-buffer.
Read back all pixels of the cleared color buffer and verify every pixel matches the specified fill color.
Read back all Z-buffer locations — including tiles not yet written after the clear — and verify every effective Z value is 0xFFFF (far-plane depth).
Verify that the displayed front-buffer is unmodified during the clear operation.

- [ ] Color buffer MEM_FILL writes the fill value to every specified SDRAM word
- [ ] Z-buffer MEM_FILL resets all Hi-Z metadata entries to the sentinel value (UNIT-005.06) and sets all Z-cache uninitialized flags to 1 (UNIT-012)
- [ ] After Z-buffer clear, any tile not written by the current frame returns 0xFFFF on first Z-read (lazy-fill via uninitialized flag in UNIT-012)
- [ ] The two-phase Z fast-clear completes without writing to individual Z-buffer SDRAM words
- [ ] FB_CACHE_CTRL.INVALIDATE_TRIGGER drops all valid and dirty bits in UNIT-013 and resets the color-cache uninitialized-flag array
- [ ] After color-cache invalidation (with black clear), first-touch writes proceed as lazy-fill with zero — no SDRAM read-fill burst required
- [ ] After color-cache invalidation followed by a non-zero MEM_FILL, subsequent rendering reads the fill color on the first cache miss
- [ ] Front-buffer is unmodified during a back-buffer clear

## Notes

The driver layer computes FILL_BASE and FILL_COUNT from the active framebuffer base address and surface dimensions defined in FB_CONFIG (INT-010): `FILL_COUNT = (1 << WIDTH_LOG2) * (1 << HEIGHT_LOG2)`.
For example, a 512×512 surface (WIDTH_LOG2=9, HEIGHT_LOG2=9) gives FILL_COUNT = 262,144 words per buffer; a 256×256 surface (WIDTH_LOG2=8, HEIGHT_LOG2=8) gives 65,536 words.
Triangle-based clearing remains available for partial clears, gradient fills, or pattern fills where a constant fill value is insufficient.

The Z fast-clear path applies to Z-buffer clears (fill value 0xFFFF targeting the Z-buffer SDRAM region).
The color-cache invalidation path applies to color-buffer clears and is triggered via FB_CACHE_CTRL.INVALIDATE_TRIGGER (INT-010).
For black clears, INVALIDATE_TRIGGER alone is sufficient — no MEM_FILL is required.
For non-zero color clears, the driver issues MEM_FILL first (to establish the fill color in SDRAM), then INVALIDATE_TRIGGER (to drop stale cached content and reset the uninitialized-flag array).
The lazy-fill contract means that SDRAM Z-buffer words for tiles not accessed after a clear may remain stale; correctness depends only on the Z-cache uninitialized flag (UNIT-012) and the Hi-Z metadata sentinel (UNIT-005.06) being consistently reset by the clear operation.
See UNIT-005.06 (Hi-Z metadata RAM), UNIT-012 (Z-buffer tile cache and uninitialized flag EBR), and UNIT-013 (color tile cache and uninitialized flag EBR) for implementation details.
