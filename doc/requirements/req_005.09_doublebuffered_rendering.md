# REQ-005.09: Double-Buffered Rendering

## Requirement

When the host submits a completed frame, the system SHALL swap the front and back framebuffer pointers in the GPU register map (INT-010) such that the display controller begins scanning out the newly rendered buffer on the next vsync boundary, while the previously displayed buffer becomes the new back-buffer available for rendering.

When the host writes FB_CACHE_CTRL.FLUSH_TRIGGER (INT-010, index 0x45) before swapping framebuffer pointers, the system SHALL write back all dirty color tile cache lines (UNIT-013) to SDRAM and block the GPU command pipeline until the flush completes, guaranteeing that the newly presented buffer in SDRAM reflects the complete rendered frame.

When the host retargets the back-buffer by writing a new address to FB_CONFIG.COLOR_BASE, the system SHOULD write FB_CACHE_CTRL.INVALIDATE_TRIGGER (INT-010) to drop cached lines from the previous back-buffer address and reset the color-cache uninitialized-flag array, preparing the cache for lazy-fill on the new back-buffer.

## Rationale

Double buffering eliminates tearing by ensuring the display always reads from a completed frame while the host writes into a separate buffer.
Swapping on a vsync boundary guarantees the transition occurs during the blanking interval, preventing a partially-updated frame from being displayed.

## Verification Method

**Test:** Render two distinct frames to alternating back-buffers.
After each swap sequence, verify that the display base address register in INT-010 points to the buffer that was most recently completed.
Verify that the previous front-buffer address is now the active back-buffer address.
Verify that swaps occur on vsync boundaries by observing that no swap takes effect mid-scan.
Verify that the display controller reads using the FB_DISPLAY.FB_WIDTH_LOG2 value written during the swap, not a stale cached value from the previous buffer configuration.
Verify that issuing FB_CACHE_CTRL.FLUSH_TRIGGER before the FB_DISPLAY write causes all dirty cache lines to appear in SDRAM before the display scan begins from the new front-buffer address.

## Notes

The two framebuffer regions are defined in INT-011 (SDRAM Memory Layout).
The framebuffer swap involves updating FB_DISPLAY.FB_ADDR and FB_DISPLAY.FB_WIDTH_LOG2 to point to the completed buffer; FB_CONFIG.COLOR_BASE is updated to the new back-buffer address for subsequent rendering.

**Required driver swap sequence:**
The driver MUST issue FB_CACHE_CTRL.FLUSH_TRIGGER and wait for completion BEFORE writing FB_DISPLAY to present the completed frame.
Without this flush, dirty tiles in the color tile cache (UNIT-013) may not yet be written to SDRAM at the time the display controller begins scanning from the new front-buffer address, producing visible tearing or stale-content artifacts.
After updating FB_CONFIG.COLOR_BASE to retarget the new back-buffer, the driver SHOULD issue FB_CACHE_CTRL.INVALIDATE_TRIGGER to drop any cached lines from the previous back-buffer and enable lazy-fill for the upcoming frame.
The complete double-buffer swap sequence is therefore: FLUSH_TRIGGER → wait → FB_DISPLAY write → FB_CONFIG.COLOR_BASE write → INVALIDATE_TRIGGER → wait.
